# Amazon S3 (Simple Storage Service) Deep Dive

This module covers the advanced architectural internals of Amazon S3, focusing on partitioning limits, encryption models, consistency, and compliance features frequently tested in senior system design interviews.

---

## 1. Under the Hood: Prefix Partitioning & Performance Limits

While S3 is marketed as having "unlimited" capacity and scale, it has physical scaling limits under the hood. Understanding S3 partitioning is crucial for high-throughput designs.

### A. Prefix Definition
S3 is a flat key-value store, not a hierarchical file system. The slashes (`/`) in a key name (e.g., `s3://my-bucket/year/month/day/file.jpg`) are just string characters, not physical folders. S3 uses these strings to partition data.
*   **Prefix**: Any string between the bucket name and the object name. In the example above, `/year/month/day/` represents the prefix.

### B. High-Throughput Request Limits
S3 automatically scales partition capacity based on request rates. However, **per prefix**, S3 imposes physical rate limits:
*   **5,500 GET/HEAD** requests per second.
*   **3,500 PUT/POST/DELETE** requests per second.
*   *Interview Impact*: If your application reads 20,000 files/sec from a single prefix (e.g., `/images/`), S3 will return `HTTP 503 Slow Down` errors.

### C. Scaling Throughput (Prefix Partitioning Strategy)
To bypass the per-prefix limit, you must distribute your keys across multiple distinct prefixes. S3 allocates a physical partition for each prefix as request rates scale.
*   *Bad Design*: Sequential dates or incremental IDs.
    *   `s3://my-bucket/2026-06-01/data-1.json`
    *   `s3://my-bucket/2026-06-01/data-2.json` (all hits go to the same partition)
*   *Good Design*: Add a hash or random character prefix.
    *   `s3://my-bucket/a8f3/2026-06-01/data-1.json`
    *   `s3://my-bucket/c9d2/2026-06-01/data-2.json` (hits are distributed across different physical partitions)

## 2. Advanced Performance Optimization

### A. Multipart Uploads
*   **How it Works**: Splitting a single large object into multiple parts and uploading them concurrently. Once all parts are uploaded, S3 reassembles them.
*   *When to Use*: **Required for objects larger than 5 GB**. Recommended for any object larger than 100 MB.
*   *Benefits*:
    *   *Resilience*: If a single network connection drops, only that specific part fails and needs to be re-uploaded.
    *   *Throughput*: Uploading multiple parts in parallel utilizes more network bandwidth.
    *   *Resume/Pause*: Uploads can be paused and resumed.
*   *Crucial Cleanup*: Always configure an **S3 Lifecycle Rule** to abort incomplete multipart uploads (`AbortIncompleteMultipartUpload`). Otherwise, orphan parts are stored in S3, and you continue to pay for them.

### B. S3 Select & Glacier Select
*   **Core Concept**: S3 Select allows applications to retrieve only a subset of data from an object using simple SQL statements (`SELECT * FROM s3object s WHERE s.age > 21`).
*   *Formats Supported*: CSV, JSON, Parquet.
*   *Benefit*: Instead of downloading a 5 GB file to query one row (consuming bandwidth and memory on your app server), S3 performs the query at the storage layer and only returns the matching rows (often reducing data transfer by 99%).

## 3. Data Protection & Compliance

### A. Object Lock (WORM Model)
S3 Object Lock lets you store objects using a Write Once, Read Many (WORM) model. Once locked, the object cannot be deleted or overwritten.
*   **Retention Modes**:
    1.  **Compliance Mode**: No one (including the AWS root user) can delete the object or alter its lock settings during the retention period.
    2.  **Governance Mode**: Users with special permissions (e.g., `s3:BypassGovernanceRetention`) can bypass the lock to delete or modify the object.
*   **Legal Hold**: Prevents deletion indefinitely until explicitly removed. Does not have a timer.

### B. MFA Delete
Provides additional protection against accidental or malicious deletions. When MFA Delete is enabled, the bucket owner must provide a hardware/virtual MFA device token to perform:
1.  Permanently deleting an object version.
2.  Suspending versioning on the bucket.
*   *Note*: Normal API deletes that place a "Delete Marker" do not require MFA. Only the permanent removal of a version requires it.

## 4. Encryption Models

AWS S3 supports multiple methods of server-side encryption (SSE) and client-side encryption.

```
+-------------------------------------------------------------+
|                      S3 Encryption Options                  |
|                                                             |
|  [Client] --- Plaintext ---> [ S3 Server ]                  |
|                                     |                       |
|        +----------------------------+-----------------+     |
|        |                            |                 |     |
|     (SSE-S3)                    (SSE-KMS)          (SSE-C)  |
|  S3 manages keys.             KMS manages keys.   Client    |
|  AWS-managed keys.            Fine-grained IAM    supplies  |
|  AES-256.                     audit trail.        key in    |
|                                                   headers.  |
+-------------------------------------------------------------+
```

### A. Server-Side Encryption (SSE-S3)
*   **Keys**: Managed entirely by AWS S3.
*   **Algorithm**: AES-256.
*   **Complexity**: Simplest (Zero overhead, enabled by default on all buckets).

### B. Server-Side Encryption with KMS (SSE-KMS)
*   **Keys**: Managed by AWS Key Management Service (KMS).
*   **Benefits**:
    *   Audit trail: CloudTrail logs every key access.
    *   Key rotation: Automated key rotation configurations.
    *   Separation of duties: A user must have S3 read permissions *and* KMS decrypt permissions to read a file.

### C. Server-Side Encryption with Customer Keys (SSE-C)
*   **Keys**: Provided by the customer.
*   **How it works**: The client passes the encryption key in the HTTP headers of the request. S3 performs the encryption in memory and discards the key.
*   *Risk*: If the customer loses the key, the data is permanently unrecoverable. AWS does not store or backup this key.

### D. Client-Side Encryption
*   **How it works**: The client encrypts the data locally before sending it to S3. S3 only sees raw, encrypted binary bytes.

## 5. Standard Interview Questions

#### Q1: You have a system processing S3 uploads via S3 Events triggering a Lambda. During peak traffic, you notice some files are uploaded but the processing Lambda does not fire. Why?
If you configured S3 to invoke Lambda directly on `s3:ObjectCreated:*` events, AWS handles this asynchronously. If your event rate exceeds the concurrency limits, Lambda will throttle. 
*   *Solution*: Place an **Amazon SQS Queue** or **Amazon SNS Topic** between the S3 event and the Lambda. S3 sends events to SQS, which buffers the messages. Lambda reads from SQS, ensuring no events are dropped.

#### Q2: What is an S3 Delete Marker? How does it relate to Versioning?
When versioning is enabled on a bucket:
*   A `DELETE` request does not actually delete the object. Instead, S3 inserts a **Delete Marker** as the current version of the object.
*   A `GET` request on the object will return an HTTP 404 (Not Found).
*   To permanently delete the object, you must specify the exact `VersionId` of the object to delete it, or remove the Delete Marker itself to restore the object.

#### Q3: How does SSE-KMS impact cost and performance during high-volume reads/writes in S3?
Every write (PUT) or read (GET) of an SSE-KMS encrypted object requires a call to the KMS service (`GenerateDataKey` or `Decrypt`).
*   *Cost*: KMS API calls are billed per 10,000 requests, which can dramatically spike your AWS bill.
*   *Performance*: KMS has rate limits (cryptographic requests per second, e.g., 10,000 request/sec). If exceeded, S3 requests will fail with KMS throttling errors.
*   *Mitigation*: Enable **S3 Bucket Keys** (configured in bucket properties). This caches file-level keys within S3, reducing the calls to KMS by up to 99%.
